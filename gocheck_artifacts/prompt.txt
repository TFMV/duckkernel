You are a senior systems engineer designing the Execution Planner and Cache Coordinator for a system called DuckKernel.

This component is responsible for:

transforming DAG recompute plans into efficient execution strategies, minimizing work via caching, and coordinating dataset materialization.

This is the layer that turns a DAG into actual execution decisions.

🎯 Purpose

Given:

a DAG of dataset nodes
node metadata (status, version, cache state)
a target dataset to compute

The system must:

Determine which nodes must be recomputed
Skip nodes that are already valid and cached
Produce a minimal execution plan
Coordinate execution order
Update cache and node state after execution
🧱 Core Responsibilities
1. Execution Planning

Input:

target_node_id


Output:

ExecutionPlan


Where:

type ExecutionPlan struct {
    Nodes []PlannedNode // ordered
}


Each node must include:

type PlannedNode struct {
    NodeID      string
    SQL         string
    Dependencies []string
    Cached      bool
    Action      ExecutionAction
}

2. Execution Actions

Define:

type ExecutionAction string

const (
    ActionSkip        ExecutionAction = "skip"
    ActionCompute     ExecutionAction = "compute"
    ActionMaterialize ExecutionAction = "materialize"
)


Rules:

Skip if node is clean and cached
Compute if dirty
Materialize if configured (persist result)
3. Cache Coordination

Implement a cache layer:

type Cache interface {
    Has(nodeID string, version string) bool
    Get(nodeID string, version string) (ResultRef, error)
    Store(nodeID string, version string, result ResultRef) error
    Invalidate(nodeID string)
}


Requirements:

version-aware caching
deterministic cache keys
safe invalidation
4. Dirty State Evaluation

Each node has:

type NodeStatus string

const (
    StatusClean NodeStatus = "clean"
    StatusDirty NodeStatus = "dirty"
)


Rules:

Node is dirty if:
SQL changed
any upstream dependency changed
Dirty state propagates via DAG
5. Minimal Recompute Strategy

Planner must:

traverse DAG upstream from target
identify minimal set of nodes requiring recompute
skip entire clean subgraphs

This is critical:

never recompute nodes unnecessarily

6. Execution Ordering

Must produce topologically sorted execution order:

dependencies always execute before dependents
deterministic ordering required
7. Execution Coordination (no actual SQL execution here)

Planner does NOT execute SQL.

It produces a plan for an executor.

8. State Updates

After execution (simulated or real), planner must support:

MarkComplete(nodeID string)


Which:

marks node as clean
updates cache state
records execution metadata
9. Failure Handling (basic)

If a node fails:

mark node as failed
downstream nodes must not execute
planner must allow retry
10. Concurrency Model

Planner must be safe under concurrent requests:

multiple planning requests allowed
execution coordination must avoid race conditions on node state
11. Determinism Requirement

Same DAG state must always produce:

identical execution plan
identical node ordering
identical cache decisions
12. Integration Points

Planner depends on:

type DAG interface {
    GetNode(id string) (Node, error)
    GetUpstream(id string) []Node
}


Planner must NOT mutate DAG structure directly.

13. Package Structure
/internal/execution/planner
/internal/execution/cache
/internal/execution/strategy
/internal/execution/state

🚫 Explicit Constraints
Do NOT execute SQL
Do NOT embed DuckDB logic
Do NOT couple tightly to CLI
Do NOT introduce distributed execution
Do NOT over-engineer cache (in-memory OK)
🧠 Design Philosophy

This system is:

a query optimizer + execution scheduler for a DAG-based compute engine

It must behave like:

Spark planner (simplified)
database optimizer (minimal)
cache-aware execution engine
📦 Output Requirements

Generate:

1. Full Go implementation

Include:

planner
cache interface + in-memory implementation
execution plan builder
dirty state evaluator
2. Example scenario

DAG:

users → active_users → enriched_users → analytics


Cases:

Case 1: All clean

→ entire plan = skip

Case 2: users updated

→ recompute all downstream

Case 3: active_users cached

→ skip active_users, recompute downstream only

3. Execution plan output example
[
  {"node": "users", "action": "compute"},
  {"node": "active_users", "action": "skip"},
  {"node": "enriched_users", "action": "compute"},
  {"node": "analytics", "action": "compute"}
]

4. Minimal test cases
clean graph
partial invalidation
full recompute
cache hit/miss scenarios
🧭 Success Criteria

The system is correct if:

recompute is always minimal
cache is always respected
execution order is correct
no redundant computation occurs
deterministic plans are produced
🔥 Why this layer matters

This is where DuckKernel stops being:

a graph tracker
a SQL wrapper

and becomes:

a real compute engine with intelligence about work

You are a senior systems engineer designing the Execution Runtime for a system called DuckKernel.

This component is responsible for:

executing planned dataset transformations using DuckDB, streaming results, and handling materialization and persistence.

This is the lowest-level execution layer of the system.

🎯 Purpose

Given:

an ExecutionPlan (ordered nodes with actions)
access to DAG metadata
SQL definitions per node

The runtime must:

Execute SQL transformations using DuckDB
Respect execution order
Stream results when appropriate
Materialize datasets when required
Update cache and execution state
Handle failures safely
🧱 Core Responsibilities
1. DuckDB Adapter

Implement a wrapper around DuckDB:

type DuckDB interface {
    Exec(ctx context.Context, sql string) error
    Query(ctx context.Context, sql string) (Rows, error)
    Stream(ctx context.Context, sql string) (RecordStream, error)
}


Requirements:

Use embedded DuckDB (Go bindings)
Maintain a persistent connection
Support concurrent-safe usage (serialize if needed)
Provide context-aware execution (timeouts, cancellation)
2. Execution Engine

Main entrypoint:

func ExecutePlan(ctx context.Context, plan ExecutionPlan) error


Behavior:

iterate over nodes in order
for each node:
skip if action = skip
execute if action = compute
persist if action = materialize
3. SQL Execution Strategy

For each node:

Compute:
CREATE OR REPLACE TEMP TABLE <node_id> AS <sql>

Materialize:
CREATE TABLE <node_id> AS <sql>


OR optionally:

COPY (SELECT * FROM <node_id>) TO '<path>.parquet'

4. Streaming Support

Implement streaming interface:

type RecordStream interface {
    Next() bool
    Record() any
    Err() error
    Close() error
}


Runtime must support:

StreamNode(ctx, nodeID string) (RecordStream, error)


Use cases:

large result sets
downstream processing
CLI preview
5. Dataset Namespace Management

All datasets must exist in a controlled namespace:

temp tables for intermediate nodes
persistent tables for materialized nodes

Naming convention:

dk_<node_id>


Must avoid:

collisions
leaking stale tables
6. Materialization Modes

Each node has:

type MaterializationMode string

const (
    Ephemeral   MaterializationMode = "ephemeral"
    Cached      MaterializationMode = "cached"
    Persistent  MaterializationMode = "persistent"
)


Behavior:

Ephemeral → temp table only
Cached → temp + cache reference
Persistent → durable table or file
7. Cache Integration

After successful execution:

cache.Store(nodeID, version, resultRef)


ResultRef may include:

table name
file path
metadata

On skip:

runtime must ensure dataset exists in DuckDB context
8. Failure Handling

If execution fails:

stop execution immediately
mark node as failed
do NOT execute downstream nodes
return detailed error

Include:

SQL string
node ID
error message
9. Transaction Model

For MVP:

each node executes independently

Optional enhancement:

wrap entire plan in transaction (if feasible)
10. Concurrency Model

Execution is:

sequential per plan (required for correctness)
but multiple plans may run concurrently

Must ensure:

DuckDB access is safe (mutex or connection pool)
no table name conflicts across runs
11. Observability

Must include:

structured logs per node:
start
SQL executed
duration
result rows (optional)
debug mode prints SQL
12. Cleanup Strategy

After execution:

drop ephemeral tables if no longer needed
retain cached or persistent datasets

Provide:

Cleanup(nodeID string)

13. Integration Points

Runtime integrates with:

Execution Planner (input)
Cache (store/retrieve)
DAG (state updates)

Must call:

MarkComplete(nodeID)


after successful execution

14. Package Structure
/internal/execution/runtime
/internal/execution/duckdb
/internal/execution/stream
/internal/execution/materialize

🚫 Explicit Constraints
Do NOT implement distributed execution
Do NOT over-engineer storage layer
Do NOT expose DuckDB internals outside adapter
Do NOT mix planning logic here
🧠 Design Philosophy

This system is:

a deterministic execution runtime over a relational compute graph

It must be:

simple
predictable
observable
correct
📦 Output Requirements
1. Full Go implementation

Include:

DuckDB adapter
execution loop
streaming support
materialization logic
cache integration
2. Example execution

Given plan:

users → active_users → enriched_users


Show:

SQL executed per node
table creation
materialization behavior
3. Streaming example
stream, _ := runtime.StreamNode(ctx, "analytics")


Iterate results safely

4. Failure scenario
simulate SQL error
show runtime halting execution
error propagation
🧭 Success Criteria

The runtime is correct if:

execution order is respected
DuckDB executes all SQL correctly
streaming works without memory explosion
materialization is consistent
failures are safe and contained
